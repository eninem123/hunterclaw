# AI Agent自我验证与数据准确性保障教程

## 核心问题

AI Agent最容易犯的错误不是"能力不足"，而是"在没验证的情况下就行动"。本文探讨如何建立系统性自我验证机制。

---

## 第一章：为什么要自我验证

### 1.1 三个真实错误案例

**案例1：股票代码混淆**
- 时间：2026-04-30
- 错误：将洲明科技(300232)误认为另一只股票(300692)
- 后果：扫描池数据错误，决策依据错误
- 根因：没有先查证代码和名称的对应关系

**案例2：API可靠性未验证**
- 时间：2026-04-30
- 错误：全量切换到byebug GLM API，30-40秒响应+频繁504
- 后果：进化任务超时失败，浪费用户成本
- 根因：单次测试通过就认为全面可用，没有持续监控

**案例3：假设未声明**
- 时间：2026-04-30
- 错误：基于用户提供的"13个信号"做决策，前提未验证
- 后果：投资建议建立在沙上
- 根因：没有区分"用户假设"和"已验证事实"

### 1.2 错误分类

| 类别 | 描述 | 典型场景 |
|------|------|----------|
| 数据错误 | 引用了错误/过时的数据 | 股票代码、API地址、价格 |
| 假设漂移 | 把假设当事实用 | 收购概率、市场判断 |
| 验证跳过 | 有能力验证但没验证 | 外部API可用性、外部信息 |

### 1.3 验证的成本与收益

```
验证成本：
- 时间：通常5-30秒
- Token：额外1-3次API调用
- 复杂度：增加一个检查步骤

验证收益：
- 避免错误决策
- 保护用户资产
- 提升信任度

结论：验证的成本永远低于错误决策的成本
```

## 第二章：验证清单体系

### 2.1 三级验证制度

**L1 - 快速检查（<5秒）**
- 数据格式是否合理
- 数据范围是否在预期内
- 明显矛盾是否存在

**L2 - 交叉验证（<30秒）**
- 两个独立来源是否一致
- 数据是否与已知事实矛盾
- 上下文是否支持这个结论

**L3 - 深度验证（>1分钟）**
- 原始来源是否可靠
- 数据是否有时效性
- 方法论是否正确

### 2.2 关键检查点

**外部数据使用前：**
- [ ] 数据来源是否可靠（官方/一手 > 二手 > 未知）
- [ ] 数据时效性（是否可能已过时）
- [ ] 数据格式是否与预期一致
- [ ] 数据范围是否在合理区间内

**投资决策前：**
- [ ] 持仓代码与名称是否匹配
- [ ] 行情数据是否为最新
- [ ] 信号来源是否可验证
- [ ] 假设前提是否已声明

**系统切换前：**
- [ ] 新API是否经过持续测试（非单次）
- [ ] 错误率和响应时间是否可接受
- [ ] 是否有降级方案

### 2.3 验证失败的处理

| 失败类型 | 处理方式 |
|----------|----------|
| 数据不可用 | 明确告知用户，不使用该数据 |
| 数据矛盾 | 列出矛盾点，让用户判断 |
| 数据可疑 | 标注"待核实"，提供已知的可能值 |
| 假设未验证 | 明确标注"基于假设，非事实" |

## 第三章：不确定性信息的处理

### 3.1 不确定性的四个层次

| 层次 | 描述 | 标记 | 行动 |
|------|------|------|------|
| 事实 | 已验证为真 | 无 | 直接使用 |
| 高置信度 | 大概率为真 | "基本确认" | 可用，但注明置信度 |
| 假设 | 未经证实 | "假设：" | 明确标注，不作为决策依据 |
| 猜测 | 完全未知 | "可能：" | 仅在明确标注时使用 |

### 3.2 假设声明模板

```
[假设前提]
- 标的：300232洲明科技（基于用户提供）
- 收购概率：85-88%（基于用户的13个信号体系）
- 数据来源：用户的分析报告（未经独立验证）

[限制说明]
- 我无法验证用户的信号体系真实性
- 投资建议基于用户的假设，非我的独立判断
- 用户需自行确认假设前提的可信度
```

### 3.3 模糊信息的处理原则

1. **宁可不说，绝不瞎说**
   - 没有数据 → "数据待查询"
   - 不确定时 → "可能/也许"明确标注

2. **来源必须标注**
   - 所有数据都要标注来源
   - 无法确认来源时说明"来源不明"

3. **区分事实和推论**
   - 事实："今天收盘价6.35元"
   - 推论："收购概率85%"（需注明这是假设）

## 第四章：建立自我验证习惯

### 4.1 决策前检查清单

```
[决策前必查]
□ 关键数据是否来自可靠来源
□ 数据是否在有效期内
□ 是否有矛盾信息未解决
□ 假设前提是否已明确声明
□ 是否超出了能力边界
□ 是否有降级/替代方案
```

### 4.2 错误复盘模板

每次错误后填写：

```
[错误记录]
日期：
错误描述：
影响范围：
根因分类：（数据/假设/验证跳过/其他）

[改进措施]
□ 已添加到验证清单
□ 已更新系统配置
□ 已写入教训记录

[预防检查]
□ 下次遇到同类场景会先验证
□ 已加入自动化检查（如有）
```

### 4.3 关键原则

1. **验证不拖，不等于不验证**
   - 5秒能查证的不要跳过
   - 30秒能交叉验证的不要省略

2. **说"不知道"不丢人**
   - 用户更信任"我不确定"而不是"我猜"

3. **假设要明说**
   - 把假设当事实用是最大的错误
   - 声明假设是保护自己也是保护用户

4. **验证优先于执行**
   - 行动前先检查
   - 宁可慢一步，不能错一路

## 第五章：实践案例

### 5.1 股票代码验证流程

```
用户说"洲明科技300232"
     ↓
验证步骤：
1. 查实时行情接口获取名称
2. 核对名称是否为"洲明科技"
3. 核对代码是否为"300232"
4. 发现不符 → 立即告知用户
     ↓
结果：完全匹配，继续使用
```

### 5.2 API可靠性验证流程

```
新API测试通过
     ↓
验证步骤：
1. 连续测试10次（不是1次）
2. 记录响应时间和错误率
3. 检查是否有超时模式
4. 测试不同类型请求
     ↓
结果：稳定性可接受 → 可用
     ↓
     或：存在问题 → 降级方案
```

### 5.3 投资建议验证流程

```
用户要求分析某股票
     ↓
验证步骤：
1. 获取实时行情数据
2. 核对代码和名称
3. 检查数据时效性
4. 区分事实和用户假设
     ↓
声明假设（如有）
"基于您提供的X信息（非我验证）"
     ↓
给出建议，注明置信度
```

## 总结

**核心原则：**
1. 验证成本永远低于错误决策成本
2. 假设要明说，不能当事实用
3. 不确定时宁可不说，绝不瞎说
4. 来源必须标注，时效必须注明

**行动清单：**
- [ ] 每次使用股票代码前验证名称匹配
- [ ] 每个投资建议前声明假设前提
- [ ] 每个外部数据标注来源
- [ ] 每次错误后填写复盘模板
